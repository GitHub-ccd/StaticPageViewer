T2DM Model 1 Technical Reference for Clinical stakeholders

Title and Overview

T2DM Model 1 — Unplanned Admission Risk Prediction

Purpose: Predict which Type 2 Diabetes Mellitus patients are at highest risk of unplanned hospital admission (including observation stays) or ED visit in the next 12 months, to prioritise patients for care management interventions.

Review context: This June 2026 technical reference summarises the current model direction for clinical stakeholders using the latest validated results and the integrated all-11 feature design.

Patient Cohort

1. Patient Cohort

Total patients: 299,224

1.1 Inclusion Criteria

Patients meeting ANY of:
* Active on the Diabetes Registry (Registry ID 82000, Active status within feature window), OR
* Two HbA1c values ≥ 6.5% (and < 20) separated by 90+ days within the feature window

1.2 Exclusion Criteria

  • Major pancreatic surgery (total/partial pancreatectomy — 6 specific Epic procedure codes)
  • Deceased before feature end date (includes PAT_STATUS_C-based death flag for out-of-system deaths)
  • Not empaneled — no active PCP assignment overlapping the feature window (EFF_DATE ≤ feature end, TERM_DATE ≥ feature start). This condition filters for patients with a current primary care provider, i.e., empaneled patients only.
  • Age outside 18–100 at feature end date

1.3 Time Windows

Window Period Purpose
Feature window May 2022 – March 2024 (22 months) Clinical history used to construct features
Target window March 2024 – March 2025 (12 months) Future unplanned admission outcome to predict

Target Variable

2. What We Are Predicting

Binary outcome: Will this patient have ANY unplanned admission (inpatient or observation) or ED visit in the next 12 months?

Metric Value
Total patients 299,224
Patients with unplanned event (positive class) 22,496
Positive rate 7.52%

This is a class-imbalanced problem — most patients do not have an unplanned event. The model is therefore judged primarily on how well it identifies the minority at-risk group without overwhelming care managers with low-yield review lists.

Why Re Engineer Features

3. What Prompted the Model Improvements

The initial model established a working baseline for risk stratification across the T2DM population, but chart review and model review showed opportunities to improve both clinical interpretability and capture of multi-morbid patients.

3.1 Chart Review Findings

The model aligned well for many patients, but some clinically complex patients appeared under-scored relative to chart-based judgment. Review suggested that the issue was not lack of data volume, but how the data had been represented.

3.2 Main Feature Representation Gaps

Finding Clinical Impact
Features often reflected database structure rather than clinical concepts Harder for care managers to understand why a patient was flagged
Diagnoses were frequently sourced in isolation Less ability to distinguish a confirmed chronic condition from a one-off coding artifact
Individual medications were over-granular Treatment complexity was harder to interpret than grouped regimen patterns
Multi-source burden signals were under-represented Clinically complex patients could be under-captured

3.3 Net Result of the Redesign

The redesigned model now uses ~280 clinically organised features across 11 dimensions instead of more than 1,000 opaque wide-table features. The current integrated H2O model achieves test PR-AUC 0.387, which now exceeds the old baseline reference of 0.382 while preserving clinical interpretability and reducing feature count by about 74%.

This changes the review narrative. The project is no longer presenting a trade-off between performance and interpretability. The current evidence supports both stronger discrimination and a more clinically understandable feature set.

The integrated model now exceeds the old baseline while remaining much smaller and more clinically understandable. 8.3 Threshold Adjustability

The threshold is adjustable based on operational capacity.

  • Raising the threshold reviews fewer patients but increases yield.
  • Lowering the threshold captures more admissions but expands the review list.
  • The 0.187 threshold is therefore best understood as a balanced starting point for consultation planning, not as a fixed clinical rule.
Operational Considerations

9. Operational Considerations

The current scoring workflow uses the production-oriented prediction pipeline with feature-contract validation. This helps ensure that the assembled feature set matches what the trained model expects and prevents silent drift during scoring.

Key operational points:
* The current model direction is the all-11 integrated design, not Tier A.
* Risk scores can be surfaced in a dashboard for chart review and prioritisation.
* Patient-level explanation outputs remain available so care managers can see the main factors behind a score.
* Thresholds can be tuned to fit review capacity without changing the underlying feature design.

Limitations and Next Steps

10. Limitations and Next Steps

Current Limitations

  • SDOH screening completion remains low, limiting contribution from those features for much of the population.
  • eGFR and some laboratory naming gaps still reduce complete kidney-function capture.
  • Blood-pressure signal is now partially represented through summary inpatient systolic/diastolic features, but fuller BP parsing and trajectory logic remain a backlog item.
  • Some dimensions now contain more integrated features than their original standalone benchmark tables, so the ranking counts and the final integrated inventory should be interpreted as related but not identical views of the same model.
  • Retrospective validation is complete, but prospective validation is still needed.

Current Project Status

  • Hybrid feature integration is complete.
  • Tier A enrichment has been tested and closed.
  • The integrated all-11 model is the current best direction.
  • The feature inventory in Sections 5–7 reflects the current June 2026 integrated design.

Next Steps

  1. Run prospective validation on a newer cohort.
  2. Deploy scores and explanations into the clinical dashboard workflow.
  3. Calibrate thresholds to realistic care-management capacity.
  4. Continue targeted data-quality fixes for BP parsing, lab naming, and tobacco-specific diagnosis logic.
