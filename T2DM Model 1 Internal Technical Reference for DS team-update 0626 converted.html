T2DM Model 1 — Unplanned Admission Risk Prediction

Data Science Internal Technical Reference

Purpose: Predict which Type 2 Diabetes Mellitus patients are at highest risk of unplanned hospital admission (including observation stays) or ED visit in the next 12 months, to prioritise patients for care management interventions.

Patient Cohort

1. Patient Cohort

Total patients: 299,224

1.1 Inclusion Criteria

Patients meeting ANY of:
* Active on the Diabetes Registry (Registry ID 82000, Active status within feature window), OR
* Two HbA1c values ≥ 6.5% (and < 20) separated by 90+ days within the feature window

1.2 Exclusion Criteria

  • Major pancreatic surgery (total/partial pancreatectomy — 6 specific Epic procedure codes)
  • Deceased before feature end date (includes PAT_STATUS_C-based death flag for out-of-system deaths)
  • Not empaneled — no active PCP assignment overlapping the feature window (EFF_DATE ≤ feature end, TERM_DATE ≥ feature start). This condition filters for patients with a current primary care provider, i.e., empaneled patients only.
  • Age outside 18–100 at feature end date

1.3 Time Windows

  • Feature window: 22 months (May 2022 – March 2024) — clinical data used to build patient profiles
  • Target window: 12 months following feature end date (March 2024 – March 2025) — period in which we observe whether the unplanned event occurs
Target Variable

2. What We Are Predicting

Binary outcome: Will this patient have ANY unplanned admission (inpatient or observation) or ED visit in the next 12 months?

Metric Value
Total patients 299,224
Patients with unplanned event (positive class) 22,496
Positive rate 7.52%

This is a class-imbalanced problem — the vast majority of patients (92.5%) do NOT have an unplanned event. The model must be specifically tuned to identify the minority at-risk group without overwhelming care managers with false positives.

Why Re-Engineer Features

3. Why Re-Engineer? Old Method vs. Current Approach

The previous modelling approach (baseline) used a monolithic extraction that produced 1,069 features from a single wide CSV pipeline. It established a strong reference point (PR-AUC 0.3816) but carried important technical and clinical limitations that motivated a full redesign.

3.1 Old Method — What We Had

Property Old Method
Feature count 1,069 (across 301,443 patients)
Source Single CSV pipeline (opaque extraction, limited documentation)
Granularity Individual drug names, individual ICD codes, individual SDOH items, and 118 raw demographic OHE columns
Diagnosis sourcing Single-source flags (Problem List OR Ambulatory DX — no cross-validation)
Sparsity Many features <1% prevalence (rare drugs, rare codes) — noise-prone
Reference performance PR-AUC 0.3816 / ROC-AUC 0.845

3.2 Problems with the Old Method

  1. Black-box features, limited clinical narrative. Individual medication names and single ICD flags did not translate cleanly into care-management reasoning.
  2. No multi-source diagnostic validation. Diagnoses from Problem List, Medical History, Ambulatory DX, and Hospital DX were not reconciled into a certainty signal.
  3. Extreme sparsity and noise. Hundreds of low-prevalence features added complexity without consistent generalisable signal.
  4. No clinical grouping hierarchy. Therapeutic intent and regimen complexity were fragmented across many single-drug indicators.
  5. Limited trajectory framing. Point-in-time values dominated over trends such as A1C change, BMI change, visit recency, and longitudinal burden.
  6. Unmaintainable pipeline. A monolithic extraction made targeted refreshes and source-level auditing difficult.

3.3 Current Approach — What We Built

Property Current Method
Feature count 285 raw joined columns → 283 model features across 11 clinically organised dimensions
Source 11 independent, documented FE notebooks with persisted Delta outputs in bsmh_ds_ph_wip
Granularity Clinical groupings mapped to how clinicians reason and intervene
Diagnosis sourcing 4-source consolidation with N_SOURCES_* and related burden signals
Hybrid integration 39 gap-analysis features integrated natively into DX, Medications, and Surgical History; standalone supplement retired
Canonical all-11 result H2O integrated PR-AUC 0.387 (test), with XGBoost integrated PR-AUC 0.373 confirming reproducible signal

3.4 Design Principles

  1. Clinically interpretable by design. Every retained feature should map to a recognisable clinical concept or utilisation pattern.
  2. Multi-source validation. Diagnoses confirmed across multiple Epic sources are more trustworthy than single-source chart artifacts.
  3. Trajectory over snapshot. Trend and recency features remain central where signal exists.
  4. Hierarchical medication architecture. Therapeutic classes, diabetes-specific treatments, and regimen-complexity summaries are represented explicitly.
  5. Lift-aware feature engineering. Rare categories are not allowed to dominate the model on small-sample noise.
  6. Modular Delta architecture. Each dimension can be refreshed independently and reassembled via LEFT JOIN on PAT_ID.

3.5 What Changed Since the May Review

  • Hybrid integration is complete. The supplement experiment validated meaningful signal from Hospital DX, Medical History, Problem List, Pharm Subclass, and surgery-type features, and that signal is now embedded directly in the 11-dimension tables.
  • Tier A enrichment is closed. TE/WoE/interactions/clusters were tested and rolled back after failing to improve test PR-AUC.
  • All-11 is now the lead direction. The best current story is no longer “acceptable trade-off”; it is that the integrated all-11 design now exceeds the old baseline in H2O while remaining clinically interpretable and far smaller.

3.6 What Was Gained vs. Lost

Aspect Gained Lost
Interpretability Every feature has a clinical narrative and documented derivation Granular drug-level memorisation from the old wide table
Maintainability 11 independent notebooks, modular refresh, clear lineage Simplicity of a single opaque CSV extract
Feature efficiency ~280 features vs. 1,069 (74% reduction) Some long-tail rare signals
Diagnostic certainty Multi-source confirmation and burden counts Single-source flags that sometimes captured undocumented edge cases
Production readiness Persisted dimension tables + prediction pipeline contract checks Zero-effort CSV reload

The remainder of this document specifies each of the 11 engineered dimensions in detail — source tables, lookback windows, derivation logic, coverage, data quality, and clinical interpretation.

Known Gap: Blood Pressure Not Captured

Blood pressure (FLO_MEAS_ID = 5) is included in the extraction filter but produces no usable features. Raw BP values are stored as compound strings (e.g., 117/78, 133/67). Because the pipeline applies TRY_CAST(MEAS_VALUE AS DOUBLE), these strings return NULL and no systolic/diastolic features survive into the final table.

Future work: Parse BP strings into separate systolic and diastolic numeric columns (e.g., using SPLIT(MEAS_VALUE, '/')) to recover this clinically important signal. Hypertension is highly prevalent in this T2DM cohort, and inpatient BP instability may carry additional predictive value for unplanned readmission risk.

Coverage and Missingness

These features are populated for only about 35.6% of the cohort. Missingness is therefore a major part of the signal:

Pattern Value Interpretation
Any inpatient flowsheet coverage 35.6% Patient had at least one prior inpatient stay with recorded flowsheet vitals
No inpatient flowsheet coverage 64.4% No prior qualifying inpatient flowsheet record in the feature window
Coverage among TARGET = 0 32.5% Lower prior inpatient exposure
Coverage among TARGET = 1 74.1% Much higher prior inpatient exposure

This means the model is learning two things at once:
* Presence of inpatient vitals is a strong proxy for prior hospitalisation burden
* Within covered patients, more abnormal extrema indicate greater physiologic instability

What the Extremes Actually Capture

The features are aggregated across all inpatient flowsheet measurements in the 22-month feature window, not just the latest admission and not a time trend. For each patient, the model sees the single worst and best recorded values across all prior inpatient stays.

Typical distributions among patients with flowsheet data:
* IP_MIN_SPO2 median = 94, 1st percentile = 58
* IP_MAX_PAIN median = 8, 95th percentile = 10
* IP_MAX_PULSE median = 95, 95th percentile = 136
* IP_MAX_TEMP median = 98.6, 95th percentile = 101.0

There are a few implausible or documentation-driven extremes (for example, IP_MIN_SPO2 = 0, IP_MAX_PULSE = 498, IP_MAX_TEMP = 150), so these variables are best interpreted as capturing documented instability during prior admissions, not as precise bedside physiologic summaries.

Clinical interpretation: This dimension is partly a vital-sign-instability signal and partly a prior-inpatient-exposure signal. Patients with prior inpatient flowsheet data are already a higher-risk subset, and among them, worse oxygenation, higher pain, tachycardia, and fever-like extremes further separate those at highest risk.

6.3 Flowsheet Vitals (PR-AUC: 0.240)

Flowsheets expanded to 13 features in the refreshed run, adding averages and blood-pressure-related signal where recoverable. Coverage remains limited to the inpatient-exposed subset, but when present these features capture acuity and physiologic instability.

Source Tables

Epic Clarity Table Role
cp__ORDER_PROC Lab orders (links patient to order, carries RESULT_TIME)
cp__ORDER_RESULTS Lab results (stores ORD_VALUE as string, links to COMPONENT_ID)
cp__ORDER_STATUS Order status tracking (filters to finalized results)
cp__CLARITY_COMPONENT Component master (maps COMPONENT_ID → COMMON_NAME)

Feature Derivation

Aggregation method: MAX_BY(ORD_VALUE, RESULT_TIME) — takes the most recent value per lab component per patient. This is a single point-in-time snapshot, NOT an average or trend.

Numeric conversion: TRY_CAST(most_recent_value AS DOUBLE) — non-numeric values (e.g., ">60", "<0.01", text qualifiers) are silently dropped.

Status filters:
* CONTACT_TYPE_C = 2 — Result contact (ensures we're looking at result events, not order or specimen contacts)
* LAB_STATUS_C = 3 — Final result only (excludes In Process, Preliminary, Edited, and Edited-Final statuses)

Important: Extended Look-Back Window

The lab dimension uses a 5-year look-back (DATEADD(DAY, -1825, feature_end_date) = March 2019 – March 2024), which is longer than the 22-month feature window used by most other dimensions. This is intentional: infrequently drawn labs (TSH, lipid panel, magnesium) may not have a result within 22 months, so the wider window ensures better coverage.

The trade-off: a patient's "most recent" TSH could be 4 years old. The model sees only the value — not when it was drawn.

Known Pattern Overlaps

  • HAS_VASCULAR_ACCESS ∩ HAS_DIALYSIS_SURGERY: The pattern '%CATHETER%' in HAS_VASCULAR_ACCESS also matches DIALYSIS_CATHETER_INSERTION and DIALYSIS_CATHETER_REMOVAL. A patient with dialysis catheters will trigger both flags.
  • HAS_CARDIAC_SURGERY includes diagnostics: EKG, ECHO, and DFB are diagnostic tests, not surgical procedures. This makes the flag a "cardiac involvement" indicator rather than a pure surgical flag.
  • HAS_GI_PROCEDURE includes THORACENTESIS: Thoracentesis is anatomically a pleural drainage procedure (respiratory), not gastrointestinal. It was grouped here during initial development due to shared procedural patterns with endoscopy. The model's learned splits remain valid since the binary flag still correlates with severity, but this affects interpretability in patient-level SHAP explanations. Planned for correction in the next model iteration (see T2DM Surgical History Feature Engineering notebook for details).

Future iteration: Apply a recency window (e.g., 3 years) to dynamic SDOH fields such as financial strain and food insecurity, while retaining the unbounded lookback for stable attributes like education and substance use history. Aggregation Logic

MAX() per patient (grouped by PAT_ID):
* For decoded string columns (ALCOHOL_USE, SMOKING_TOBACCO_USE, TOBACCO_USE): MAX returns the alphabetically last value, NOT necessarily the most recent. Example: if a patient has "Never" and "Yes" records, MAX returns "Yes".

* Categories with similar lift profiles → merged to strengthen signal (e.g., all Hispanic subcategories merged)
* Categories with distinct protective signals kept separate (e.g., "Declined" ethnicity)

8.1 Key Findings

  1. H2O integrated at 0.387 is the canonical all-11 number for this review. It exceeds the old baseline (0.382) while using roughly 74% fewer features.
  2. XGBoost integrated at 0.373 confirms the signal is real and reproducible in a second modelling framework. The hybrid integration improved the XGBoost combined run from 0.365 to 0.373.

  4. Tier A is now historical context, not the forward recommendation. It was important in May as a parsimony candidate, but the current best-performing and most defensible direction is the integrated all-11 design.
  5. Interpretability remains the differentiator. Every retained feature still maps to a chart-verifiable clinical or utilisation concept despite the stronger combined performance.

Modelling Methodology

9. Modelling Methodology

Component Approach
Algorithm XGBoost (gradient-boosted decision trees)
Hyperparameter tuning Optuna Bayesian optimisation — 50 trials per experiment
Cross-validation 10-fold stratified (preserves 7.5% positive rate in each fold)
Primary metric PR-AUC (optimised for identifying the rare at-risk group)
Secondary metrics ROC-AUC, F1, Precision, Recall
Train/Test split 70%/30% stratified random split (fixed seed for reproducibility)
Missing data handling XGBoost handles NaN natively (learns optimal split direction for missing values)
Feature explainability SHAP (SHapley Additive exPlanations) — enables individual patient-level risk factor identification
Experiment tracking MLflow — all runs logged with metrics, parameters, and SHAP artifacts

9.1 Why XGBoost?

XGBoost (eXtreme Gradient Boosting) is a tree-based ensemble algorithm that:
* Handles mixed data types (continuous labs, binary flags, categorical demographics) without preprocessing
* Naturally handles missing values — critical given our 74% eGFR missingness and 88%+ SDOH missingness
* Produces interpretable feature importance rankings
* Is well-validated in clinical risk prediction literature

Next Steps and Limitations

11. Next Steps & Limitations

11.1 Current Limitations

  • SDOH screening completion remains low (88–99% missing for screening-dependent fields), limiting how much that dimension can contribute for most patients.
  • eGFR remains missing for many patients because capture depends on nephrology-related testing pathways.
  • Flowsheet vitals cover only the inpatient-exposed subset of the cohort, so that dimension remains high-value but selective.
  • Blood pressure parsing is still incomplete — concatenated BP strings remain a recoverable signal for a future refresh.
  • Three lab component naming mismatches remain unresolved (GLOM FILT RATE EST, HEMOGLOBIN A1C (POC), TRIGLYCERIDES).
  • TOBACCO-specific DX signal is still shadowed by CASE-order logic and requires a retrain after correction.
  • Retrospective validation only. The model has not yet been validated prospectively on a newer cohort.
  • Single time-slice evaluation. Temporal stability across multiple refresh periods is still to be demonstrated.

11.2 Current Programme Status

  1. Hybrid integration is complete. The standalone supplement path has been validated, absorbed into the core feature tables, and retired.
  2. Tier A enrichment is closed. It remains part of project history, but it is not an active recommendation or future workstream.
  3. All-11 integrated is the current best model direction. For June 2026 documentation and discussion, H2O 0.387 is the canonical combined result.

11.3 Next Steps

  1. Prospective validation: Test the integrated all-11 model on a more recent cohort to confirm temporal stability and operational plausibility.
  2. Dashboard deployment for review workflows: Surface risk scores and explanations in the clinical dashboard so chart review and care-management use can be evaluated in practice.
  3. Threshold calibration: Use the consultation metrics as the starting point and tune thresholds to realistic review capacity.
  4. Prediction pipeline alignment: Keep the active scoring workflow aligned with the current all-11 integrated feature contract and champion model selection rules.
